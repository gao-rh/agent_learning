# OpenClaw 中介层讨论日志

## 场景

- 时间：2026-06-13
- 主题：OpenClaw 在 AI Agent 系统中的位置
- 来源：李宏毅机器学习 2026 第一节 PPT

## 用户假设

用户看到 PPT 后的理解：

- OpenClaw 其实是在 AIGC 体系里面最不 AI 的部分。
- 它是人和语言模型的一个介质，是帮助我们使用语言模型的中间节点。
- 严格意义上，OpenClaw 只是一个软件、一个工程，是一个拼装和智能处理的脚手架。

继续拆这个脚手架时，用户当前理解：

- 第一层可以先看 `System Prompt`。
- System Prompt 会加载 Agent 的基本信息，例如 `Soul`、`Identity`、`User`、`Memory`。
- System Prompt 会加载行为准则，例如 `AGENTS.md`。
- System Prompt 会说明 Agent 可以使用哪些 `Skill`。
- System Prompt 也会包含之前的记忆应该如何检索、如何使用等信息。

进一步讨论后，用户认为“Agent 当前回合上下文包”的摘要示例更有参考价值，因为它比单独说 System Prompt 更完整。

用户进一步理解：

- 正常和 AI 对话时，系统其实是在做上下文拼接。
- 模型不是真的在参数里记住了你之前说过什么，而是把之前的上下文拼起来，再交给大模型继续预测和生成。

继续从“上下文”进入“行动”：

- Agent 要做事情、使用电脑，核心方法是使用工具。
- 工具的信息会包含在 `System Prompt` 或当前上下文包里，让模型知道自己能用哪些动作。
- 用户进一步追问：Agent 的回复是怎么具体触发工具执行的？是回复里有特殊标识符被系统识别，还是其他机制？

## 助手反馈

初步反馈：

- 这个理解方向是有价值的，因为它把“模型智能”和“工程中介层”分开了。
- 但“OpenClaw 不是 AI”这句话可能需要更精确：它不是基础智能的主要来源，但它仍然是 Agent 系统的重要组成部分。
- 更严谨的表达可以是：OpenClaw 是把聊天渠道、记忆、任务管理、电脑操作、权限控制和语言模型调用拼装起来的 orchestration layer。
- 对 System Prompt 的理解方向也对：它可以看成 Agent 单次回合的“上下文装配层”，把身份、用户、记忆、规则、技能和检索方式组织成模型可用的指令。
- 但需要继续区分：System Prompt 不是整个 Agent Runtime。真正的脚手架还包括消息渠道、profile/workspace、模型路由、工具执行、权限控制、任务状态、记忆存储和结果回写。
- 更完整的表达是：模型每一轮真正使用的是“当前回合上下文包”，它可能由系统规则、开发者规则、项目规则、用户对话、长期记忆摘要、当前环境、已读取文件、工具结果和任务目标共同组成。
- 上下文不是记忆本身；只有被检索、读取、整理并注入进当前回合的信息，才真正成为模型可用的上下文。
- 这个理解方向是对的，但要补一个边界：产品体验里的“记住”可以由多种机制实现，包括原始对话历史、压缩摘要、外部长期记忆、项目文件和检索结果。它们通常不是即时改写模型权重，而是在运行时重新注入给模型。
- 对工具使用的理解方向也对：工具说明进入上下文，模型选择工具调用；但真正执行电脑操作的是 Agent Runtime / 工具运行时，运行时还负责权限检查、执行、失败处理和结果回传。
- 工具触发通常有两类机制：现代 API 原生结构化工具调用，模型返回 `tool_call` / `function_call` / `tool_use` 这类结构化对象；早期或简易框架可能要求模型输出特殊文本标记，再由 Runtime 解析执行。
- 无论哪种方式，模型通常只是提出工具调用和参数；真正的执行、权限检查、错误处理和结果回写由系统 / OpenClaw / Agent Runtime 完成。

## 待确认

- 用户是否认可“OpenClaw 不是智能来源，而是把模型智能工程化的中介层”这个表达？
- 是否需要把它提炼为概念卡：`Agent 中介层`？
- 是否需要补一个反例：没有 OpenClaw / Codex / Claude Code 这类中介层时，语言模型仍然聪明，但很难稳定接入真实工作流？
- 用户是否认可“System Prompt 是上下文装配层，但不是完整运行时”这个表达？
- 后续是否按 `System Prompt -> Memory -> Skills -> Tools -> Gateway -> Task Manager -> Permission -> Model Adapter` 的顺序拆 OpenClaw 架构？
- 是否需要把 `Agent 当前回合上下文包` 提炼成独立概念卡？
- 是否需要把 `上下文拼接不等于模型记忆` 提炼成独立概念卡？
- 是否需要把 `工具调用是 Agent 行动的接口` 提炼成独立概念卡？
- 是否需要实际查看 OpenClaw 的实现，确认它采用结构化 tool call、文本标记解析，还是混合机制？

## 已确认

- 讨论过程本身应该记录。
- 但助手不应直接改写用户观点并沉淀为正确结论。
- 助手应该先沟通、先教学、先指出不严谨处，等用户确认后再修改最终观点。
