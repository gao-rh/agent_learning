# 经营分析 Skill 架构讨论日志

## 场景

- 时间：2026-06-13
- 主题：经营分析类 Agent Skill 如何拆分、组织和维护
- 继续时间：用户预计下周一 / 下周二继续细看

## 用户背景

用户正在做一个经营分析能力，主要帮助运营同学做数据分析。由于涉及工作机密，暂时不能提供真实目录和具体接口内容。

## 当前功能理解

| 功能 | 说明 | 当前实现信息 |
| --- | --- | --- |
| 口径查询 | 查询指标及计算公式 | 通过“口径查询接口调用”skill 的接口描述实现；用户直接摘取接口描述，不使用完整 skill |
| 取数 | 从需求澄清到数据获取、计算和防护 | 包括澄清判断、数据获取方式确认、查询执行包、指标计算、防护/踩坑 |
| 查询执行 | 在口径、来源、查询方式确定后执行查询 | 执行包包括 Tool、CLI、MCP，会根据环境和场景选择 |
| 简单分析报告 | 读取另一个接口跑出的结果，做简单分析呈现 | 接口格式是一个链接请求，返回对应结果 |
| 专题分析 | 固定分析思路，多轮取数、判断、计算 | 当前只有“流量分析”，未来会增加多个专题；专题会反复使用取数能力 |
| 定时任务 | 定时自动执行特定分析 | 属于分析体系内，可能调用已有专题或取数流程 |

## 关键架构问题

用户原先写了很多个 skill，但 skill 调用另一个 skill 时出现路径混乱：脚本或 reference 在一个 skill 目录里，执行时却去另一个 skill 目录找文件，导致报错。

因此用户现在倾向不再跨 skill 引用，而是摘录/复制必要内容。

## 已讨论判断

### 1. 不能简单说“尽量拆”或“全部合”

Skill 是否拆分，需要同时看：

| 判断维度 | 倾向拆成多个 Skill | 倾向放在一个 Skill 内 |
| --- | --- | --- |
| 用户是否会独立触发 | 会，例如“只查口径”“只取数”“只做流量分析” | 很少单独触发，只是大流程内部步骤 |
| 输入输出是否独立 | 输入输出清楚，可单独完成 | 必须依赖其他模块中间状态 |
| 上下文是否高度共享 | 共享少 | 共享同一套口径、数据源、防护规则 |
| 工具/权限是否不同 | 权限差异大，应该隔离 | 工具和权限基本一致 |
| 更新频率是否不同 | 某模块经常独立迭代 | 一起迭代更自然 |
| 路径/资源依赖 | 能通过公共接口共享 | 拆开后会频繁跨目录引用 |
| 触发冲突风险 | 低 | 高，多个 Skill 容易同时触发 |

### 2. 当前更合理的方向

初步建议不是一个巨型 skill，也不是很多互相引用的小 skill，而是：

```text
按用户可独立触发的任务拆 Skill；
把共享接口、口径、防护、查询方式判断做成公共核心资料；
通过同步/复制机制进入各 Skill；
不要跨 Skill 相对路径引用内部文件。
```

### 3. 对当前业务的初步拆分建议

| 功能 | 是否适合独立 Skill | 理由 |
| --- | --- | --- |
| 口径查询 | 适合 | 用户会只查口径；输入输出清楚；接口独立 |
| 取数 | 适合 | 用户会只取数；流程复杂；需要单独防护 |
| 简单分析报告 | 适合或合入取数 | 如果用户常说“基于这个链接出个简单报告”，可独立 |
| 专题分析 | 适合独立 Skill | 它是多轮取数、判断、计算的 workflow |
| 定时任务 | 视复杂度 | 如果只是定时跑某专题，放 schedules；如果有独立状态/权限/失败重试，拆 Skill |

## 待深入问题

- 如果不能稳定使用 shared core 路径，如何维护“摘录复制”的一致性？
- 是否需要建立一个接口描述更新清单，记录每次从外部平台摘录的接口变更？
- `口径查询`、`取数`、`专题分析` 的触发描述应该如何写，避免误触发和互相抢任务？
- `取数` 内部的 Tool / CLI / MCP 如何决策，是否要形成一个 routing table？
- `流量分析` 专题如何引用取数能力：通过自然语言说明、统一输出格式，还是固定中间产物？
- 定时任务是 skill、reference、cron job，还是三者组合？
