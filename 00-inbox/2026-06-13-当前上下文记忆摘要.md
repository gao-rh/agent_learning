## 当前课程主线

本轮学习主题是李宏毅 2026 机器学习课程第一节中关于 “Agent 是什么” 的部分。重点不是 Agent 发展史，而是通过历史引入 Agent 当前的结构。

已经整理的主线：

1. OpenClaw 是一个 Agent 软件/工程脚手架，不是 AI 智能本身。
2. OpenClaw 是人、通讯软件、电脑、工具、记忆系统和语言模型之间的中介层。
3. Agent 当前上下文由 system/developer/project rules、历史消息、memory、文件、工具结果、当前任务等拼接而成。
4. 模型不是直接“记住”过去，而是 Runtime 把相关上下文拼接/检索后送入模型。
5. 工具调用通常是模型返回结构化 tool call，Runtime 执行工具，再把结果拼回上下文继续下一轮。
6. 一次完整任务通常包含多轮 Agent loop，而不是一次模型调用完成。

## 已完成概念：记忆搜索

记忆搜索已经作为一个完整主题整理完成。核心理解：

```text
记忆文件
-> chunk
-> 关键词索引 + 语义索引
-> 查询时关键词检索和语义检索并行召回
-> hybrid merge
-> top chunks 注入当前上下文
```

关键词检索：

| 概念 | 当前理解 |
| --- | --- |
| chunk | 定义检索和排序单位，主要价值不是提速，而是粒度和上下文质量。 |
| SQLite | 本地数据库引擎和数据库文件，用来存 chunk、索引和元数据。 |
| FTS5 | SQLite 的全文搜索模块。 |
| 倒排索引 | 提前建立 `token -> chunk_id`，查询时不用重新扫描全文。 |
| tokenizer | 决定文本和查询如何拆成 token；中文可用 trigram。 |
| trigram | 三字符滑窗切分，例如 `解剖小龙虾 -> 解剖小 / 剖小龙 / 小龙虾`。 |
| fallback 子串匹配 | FTS 不适合或失败时，用类似 `LIKE '%龙虾%'` 的朴素包含匹配补漏。 |
| BM25 | 关键词相关性排序算法，看词是否稀有、出现频率和 chunk 长度。SQLite FTS5 中 `bm25()` 分数越小越好。 |

语义检索：

| 概念 | 当前理解 |
| --- | --- |
| embedding model | 通用/第三方预训练模型，不是 OpenClaw 自己训练；聊天模型和 embedding 模型不同。 |
| vector | 文本的语义坐标。 |
| vector search | 把 query 和 chunk 都向量化后，用 cosine distance/similarity 找相近 chunk。 |
| provider | OpenAI、Gemini、Voyage、Mistral、DeepInfra、Bedrock、Ollama、Local 等。 |
| 启用条件 | 需要配置或自动检测到 embedding provider；否则只走关键词检索。 |

Hybrid merge：

```text
keyword candidates + vector candidates
-> 候选池合并
-> 按 chunk id 去重
-> 合并 textScore / vectorScore
-> 加权打分
-> 可选 temporal decay / MMR
-> 阈值过滤 + Top N
-> path + lines + snippet + score + source
-> 注入上下文
```

## 已完成概念：Skill 与 Sub-agent

Sub-agent / Session Span：

1. 子 Agent 的核心价值之一是 context partitioning，而不只是并行。
2. 主 Agent 应传递边界清晰的任务、必要上下文、工具/资源限制、返回格式。
3. 默认不鼓励子 Agent 无限递归生成子 Agent，否则可能出现“递归派活”或 undercommitment。
4. 子 Agent 与主 Agent 应有上下文隔离、工具隔离、资源隔离。

Skill：

1. Skill 是可复用能力包，包含触发条件、工作流、工具使用、references、scripts/assets。
2. 写 Skill 不应盲目微型化，也不应全部塞成巨型 Skill。
3. 判断拆分与否要看：用户可独立触发的任务边界、资源复用、路径稳定性、维护成本、工具权限边界。
4. 用户经营分析 Skill 当前主要包括：口径查询、取数、简单分析报告、专题分析、定时任务。
5. 当前倾向：多个用户可触发 Skill + 共享核心/接口描述，避免直接跨 Skill 相对路径混乱。

## 当前项目主线：经营分析 AI 知识库

用户正在规划一个用于经营分析和数据查询的业务知识库。核心不是普通文档问答，而是服务以下问题：

1. 这个指标为什么大幅下降？
2. 这个东西为什么会变成这样？
3. 我们想做得更好，能不能给点建议？
4. 如何支撑专题分析、报告问答、指标口径查询、取数/数据查询？

当前已确认情况：

| 项目 | 当前信息 |
| --- | --- |
| 业务范围 | 一个小业务方向的全部业务数据和分析知识，不再细分多个业务域。 |
| 向量知识库 | 有平台，上传内容后会自动向量化；数据库大概率是 Milvus，但未最终确定。 |
| 项目阶段 | 规划阶段，尚未实际操作。 |
| 现有系统 | 指标口径平台、数据目录、血缘系统，都有稳定 API。 |
| 查询能力 | 有多种查询方式，包括 SQL。 |
| 当前难点 | 数据多且杂；能查到表不代表表准确；核心要解决业务解释、归因和建议。 |
| 首批文档 | 近期几个月周报和报告；会议纪要从现在开始积累。 |
| 权限 | 暂时不考虑。 |
| 知识确认 | 机制还不清楚，需要后续设计。 |

当前方案判断：

```text
原始资料
-> 解析
-> 抽取指标/表/字段/SQL/案例/Playbook/事件/波动/假设/建议
-> 标准化与校验
-> 结构化知识库 + 现有指标平台 + 数据目录/血缘 + Milvus + 全文检索
-> Agent 按任务拿结构化事实和证据
```

最关键的抽取对象：

| 对象 | 含义 |
| --- | --- |
| `MetricMovement` | 指标发生了什么变化。 |
| `BusinessEvent` | 同期发生了什么业务事件。 |
| `Hypothesis` | 大家认为为什么发生。 |
| `ActionRecommendation` | 接下来建议做什么。 |
| `ReportCard` | 每篇周报/报告的结构化摘要。 |
| `MeetingCard` | 每次会议的结构化摘要。 |
| `AnalysisCase` | 历史分析案例。 |
| `AnalysisPlaybook` | 专题分析流程。 |
