# RAG 专题预留

## 场景

- 时间：2026-06-14
- 类型：课程预留 / 后续专题
- 来源：本轮关于 RAG、记忆搜索、成熟 RAG 策略的讨论

## 用户当前判断

RAG 仍然是一个非常重要的概念，后面应该作为一个专题单独学习。当前这一节课先继续回到“Agent 是什么 / OpenClaw / 龙虾”的主线。

## 助手反馈

这个判断是合理的。RAG 不应只理解为“向量数据库问答”，而应理解为“把外部知识、证据和工具检索结果接入模型生成过程”的一类系统方法。

更严格地说：

```text
记忆搜索 = RAG 的核心检索层之一，偏 Agent Memory 场景
成熟 RAG = 检索层 + 上下文组织 + 生成约束 + 引用 + 权限 + 评测 + 工具/结构化数据
```

`20-concepts/记忆搜索总览.md` 已经覆盖了 RAG 的核心骨架：

```text
chunk
+ FTS/BM25
+ embedding/vector search
+ hybrid merge
+ Top N 注入上下文
```

后续 RAG 专题需要在这个基础上继续学习：检索质量、证据组织、可靠生成、结构化工具和评测。

## 2026 年 RAG 总判断

| 问题 | 暂定结论 |
| --- | --- |
| RAG 还重要吗 | 重要，是企业知识型 AI 和私有知识问答的底座之一。 |
| 朴素 RAG 是否足够 | 不够。单纯“切 chunk -> 向量召回 -> 塞给模型”已经不是成熟方案。 |
| 当前成熟路线 | Hybrid Search + Metadata Filter + Rerank + Citation + Eval。 |
| 当前前沿路线 | Agentic RAG、Reasoning RAG、GraphRAG、Multimodal RAG、Contextual Retrieval、Late Chunking。 |
| 和记忆搜索的关系 | 记忆搜索是 RAG 的核心子集，解决“如何找回相关外部文本”。 |

## 目前最新的 RAG 策略

| 策略 | 核心思想 | 适合场景 | 成熟度 |
| --- | --- | --- | --- |
| Agentic RAG / Reasoning RAG | Agent 规划搜什么、拆问题、搜多轮、调用工具。 | 复杂问答、研究报告、经营分析、Deep Research。 | 前沿但快速进入产品。 |
| GraphRAG | 从文档抽实体和关系，建知识图谱，再用图谱辅助回答。 | 多文档全局总结、组织关系、事件脉络。 | 前沿，成本较高。 |
| Contextual Retrieval | 给 chunk 补充文档、章节、背景信息后再检索。 | 长文档、报告、政策、会议纪要。 | 实用，成熟度较高。 |
| Late Chunking | 先用长上下文 embedding 看完整文档，再形成 chunk embedding。 | 长文档上下文容易断裂的场景。 | 新但有价值。 |
| RAPTOR / Hierarchical RAG | chunk 聚类、递归摘要，形成树状索引。 | 长书、长报告、多层摘要问答。 | 中等成熟。 |
| Self-RAG / Corrective RAG | 判断是否需要检索、检索结果是否可靠，不可靠就纠正。 | 对事实性和引用要求高的系统。 | 研究味更强，思想重要。 |
| Multimodal RAG | 检索文本、图片、表格、截图、PDF layout、视频帧。 | 合同、财报、PPT、设计稿、医学/工业资料。 | 2025-2026 重要方向。 |
| Structured / Tool RAG | 不只搜文档，还查 SQL、API、指标平台、数据目录。 | BI、经营分析、数据问答。 | 企业场景非常关键。 |

## 目前最成熟的 RAG 策略

最成熟、最稳的落地路线：

```text
高质量文档解析
-> chunk + metadata
-> BM25 / sparse search
-> dense vector search
-> hybrid fusion
-> rerank
-> context packing
-> grounded generation
-> citations
-> evaluation
```

| 层 | 成熟做法 | 解决什么问题 |
| --- | --- | --- |
| 数据层 | 文档解析、OCR、表格抽取、版本、权限、metadata。 | 数据质量和可追溯。 |
| 切分层 | 语义 chunk、递归 chunk、overlap、parent-child chunk。 | 片段不能太碎也不能太粗。 |
| 检索层 | BM25 + dense vector + metadata filter。 | 精确词和语义都能搜到。 |
| 融合层 | RRF / score fusion。 | 合并关键词和向量结果。 |
| 精排层 | reranker / cross-encoder。 | 把真正相关的排前面。 |
| 上下文层 | 去重、扩展上下文、按证据排序、控制 token。 | 给模型的材料更干净。 |
| 生成层 | 只基于证据回答、引用来源、不知道就拒答。 | 降低幻觉。 |
| 评测层 | recall@k、MRR、faithfulness、answer correctness。 | 不靠感觉调参。 |

## 具体实施流程

```text
1. 收集资料
   文档、网页、PDF、会议纪要、周报、数据库 schema、API 文档

2. 清洗和解析
   去噪、OCR、表格抽取、标题层级、页码、来源、时间、作者、权限

3. 切 chunk
   按标题/段落/语义切分，保留 overlap 和 parent document 信息

4. 加 metadata
   document_id、source、date、department、topic、page、section、permission、version

5. 建多种索引
   BM25/全文索引 + dense vector index + metadata index

6. 查询预处理
   判断用户意图，必要时 query rewrite、multi-query、问题拆解

7. 多路召回
   keyword search、vector search、metadata filter、必要时 SQL/API 查询

8. 融合和重排
   用 RRF 或加权融合取 top 50，再用 reranker 精排到 top 5-10

9. 上下文组装
   去重，补充上下文，保留引用，避免塞太多无关片段

10. 生成答案
   要求模型基于证据回答，标注来源，不足时说明不确定或拒答

11. 校验
   检查答案是否被证据支持；关键场景可二次 verifier

12. 评测和迭代
   建测试集，看检索召回、引用准确率、答案正确率、幻觉率
```

## 与记忆搜索的差异

| 额外层 | 为什么重要 |
| --- | --- |
| 数据治理 | 知道资料是否权威、是否过期、谁能看。 |
| query rewrite / decomposition | 用户问题常常太口语、太复杂。 |
| rerank | 初筛结果不等于最终最相关结果。 |
| citation | 回答必须能追溯到来源。 |
| verifier | 防止模型把证据说歪。 |
| structured tools | 经营分析不能只搜文档，还要查指标、SQL、血缘。 |
| evaluation | 没评测就不知道 RAG 是否真的变好。 |

## 后续专题学习顺序

1. 成熟 RAG 主线：chunk、BM25、embedding、hybrid search、rerank、citation。
2. 检索质量增强：Contextual Retrieval、query rewrite、multi-query、metadata filter。
3. 复杂问题处理：query decomposition、agentic retrieval、tool RAG。
4. 长文档/全局理解：RAPTOR、GraphRAG。
5. 可靠性工程：Self-RAG、CRAG、RAGAS/eval、verifier。
