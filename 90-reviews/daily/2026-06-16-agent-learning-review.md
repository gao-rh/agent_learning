## 最重要的判断变化

### 1. Agent 不是“更聪明的模型”，而是“模型 + Runtime 的工作系统”

之前的有用直觉是：OpenClaw 是 AI Agent 里“不太 AI”的部分，是人和语言模型之间的中介层。

今天更稳定的表达应该是：

> Agent 的基础智能主要来自模型，但能否稳定做事，取决于 Runtime 如何组织上下文、工具、权限、记忆和任务状态。

这个变化重要，因为它会改变后续学习的拆解顺序。以后不应该只问“模型会不会”，还要问：

| 维度 | 要追问的问题 |
| --- | --- |
| 上下文 | 当前回合模型实际看到了什么？ |
| 工具 | 工具 schema 如何进入请求，结果如何回写？ |
| 权限 | 哪些动作被沙箱、审批、只读策略限制？ |
| 记忆 | 记忆是原文拼接、摘要，还是检索后注入？ |
| 自动化 | Heartbeat、Cron、Hooks、Webhook 分别由什么触发？ |

### 2. “上下文拼接不等于模型记忆”是后续理解 RAG、Agent 和长期助手的底层概念

当前最值得保留的严格表达是：

> 模型看起来记得，通常不是参数被即时改写，而是 Runtime 把历史、摘要、记忆检索结果、项目文件和工具结果重新放进了当前回合上下文。

这个判断会降低很多误解：不要把 `MEMORY.md`、SQLite 索引、embedding、聊天历史和模型参数混成一个“记忆”概念。后续学习 RAG 时，也应该从“资料如何进入上下文”出发，而不是从“向量库能不能搜到”出发。

### 3. 经营分析知识库的主问题不是 RAG，而是业务分析资产工程

经营分析项目里最关键的判断变化是：

> RAG 只能承担证据层；真正的经营分析知识库必须把分析案例、指标变化、业务事件、假设、行动建议、口径和关系做成可审核的业务资产。

这个变化重要，因为它把第一阶段路线从“搭一个文档问答系统”改成了：

```text
原始资料
-> 标准明细
-> 主题资产
-> 关系层
-> 数据服务
-> Agent 应用
```

如果不做这层抽象，Agent 很容易把会议里的猜测当事实、把周报建议当已验证结论、把检索命中当分析完成。

## 为什么这些变化重要

这些变化共同指向一个更稳定的学习方法：以后学习 Agent 不只看模型能力，而要看“信息如何被组织、约束、验证和复用”。

这和仓库里的长期学习流程是一致的：

| 工作流阶段 | 当前对应物 | 还缺什么 |
| --- | --- | --- |
| Capture | 原始课堂记录、讨论日志、RAG 专题预留 | 持续保持，不急着过度打磨。 |
| Understand | Agent 来源笔记、OpenClaw 机制拆解 | 对部分机制还要继续查实现。 |
| Retrieve | 部分概念卡已有自测问题 | 还需要定期回测，不只是写出来。 |
| Build | 暂时较弱 | 最小 Agent loop、RAG baseline、检索评估还没开始。 |
| Teach | 经营分析分享报告已经接近可分享 | 分享稿需要压缩成演示结构。 |
| Review | 本篇日评开始补位 | 后续要形成固定每日/每周复盘节奏。 |

最薄弱的一环是 `Build`。目前很多理解已经讲得清楚，但还没有用小实验验证。例如：工具调用循环、hybrid retrieval、metadata filter、rerank、结构化业务对象抽取，都可以做成小型 code lab。
