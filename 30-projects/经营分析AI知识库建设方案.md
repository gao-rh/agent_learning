# 经营分析 AI 知识库建设方案

## 背景与目标

这个知识库的核心目标不是“让 AI 能搜索公司文档”，而是支撑未来的经营分析和数据查询任务：

1. 查准指标口径、业务定义、计算公式。
2. 找到正确的数据表、字段、SQL、数据链路和历史查询方式。
3. 复用历史分析报告、周报、专题分析、会议结论和踩坑经验。
4. 帮助 Agent 完成取数、解释数据、生成分析判断和形成报告。
5. 让 AI 的回答可追溯、可校验、可维护，而不是只依赖向量相似度。

一句话：

> 这不是一个单纯的向量库项目，而是一个面向经营分析 Agent 的业务知识工程系统。

## 当前已确认情况

| 项目 | 当前信息 |
| --- | --- |
| 向量数据库 | 具体数据库还未最终确定，但大概率会使用 Milvus。 |
| 向量知识库平台 | 已有平台，只要上传内容即可自动向量化；项目还在规划阶段，尚未实际操作。 |
| 业务范围 | 不是多个大业务域，而是某一个小业务方向的全部业务数据和分析知识。 |
| 第一批服务范围 | 专题分析、报告问答、指标口径查询、取数/数据查询都需要覆盖，尤其要回答经营分析类追问。 |
| 现有指标体系 | 已有指标口径平台。 |
| 现有数据治理能力 | 已有数据目录和血缘系统，且指标口径平台、数据目录、血缘系统都有稳定 API。 |
| 现有查询能力 | 已有多种查询方式，包括 SQL。 |
| 当前主要困难 | 数据多且杂；能查到表不等于表一定准确。核心问题不是技术查表，而是业务解释、归因和建议。 |
| 首批文档范围 | 近期几个月的周报和报告；会议纪要从现在开始积累。 |
| 权限管理 | 暂时不作为第一阶段重点考虑。 |
| 知识确认机制 | 暂不明确，需要后续再设计 review / 确认流程。 |

因此，第一版方案不应该按“从零建设知识库”设计，而应该按“整合已有业务系统 + 建 AI 可用的知识抽取和交付层”设计。

典型目标问题：

| 老板/业务问题 | AI 需要的能力 |
| --- | --- |
| 这个指标为什么大幅下降了？ | 找到指标口径、历史趋势、相关维度、异常时间点、历史类似案例和可能归因。 |
| 这个东西为什么会变成这样？ | 识别业务现象、关联数据变化、找报告/会议/活动/策略变更证据。 |
| 我们想做得更好，能不能给点建议？ | 结合历史复盘、业务杠杆、可行动策略、风险和验证方案给建议。 |
| 最近业务有什么异常或机会？ | 聚合近期周报、报告、指标波动和会议纪要，形成经营观察。 |

## 核心判断

| 判断 | 说明 |
| --- | --- |
| Milvus 只是检索底座之一 | Milvus 适合存 embedding 并做相似度检索，但不应该成为唯一的知识源。 |
| 指标口径平台应作为权威源 | 既然已有指标口径平台，应优先同步/调用该平台，而不是另建一套冲突口径。 |
| 数据目录/血缘系统应作为数据资产权威源 | 既然已有数据目录和血缘系统，表、字段、上下游关系应优先引用现有系统。 |
| 会议纪要不能直接当权威事实 | 会议纪要更适合作为过程性证据，需要抽取决策、问题、行动项，并标注状态。 |
| 文档 chunk 只是入口 | 最终要沉淀成可复用的知识对象：指标、表、字段、分析方法、案例、SQL 模板、风险提示。 |
| AI 不应该只拿 raw chunk | 给 AI 的上下文应该是“结构化事实 + 检索片段 + 来源 + 权限 + 新鲜度 + 置信度”。 |
| 第一批范围较广，需要按任务拆接口 | 专题分析、报告问答、查口径、取数都要覆盖，所以不能只有一个通用 search。 |
| 核心知识不是“表在哪里”，而是“业务现象如何解释” | 数据目录解决找表，经营分析知识库要解决归因、判断、建议和复用历史经验。 |

## 总体架构

```text
原始知识源
-> 采集与解析
-> 知识抽取与标准化
-> 知识对象存储
-> 多路检索与工具调用
-> Agent 上下文组装
-> 反馈、评估、治理
```

更具体：

```text
文档/报告/会议纪要/代码/SQL/数仓元数据
        ↓
解析：文本、表格、章节、代码块、SQL、元信息
        ↓
抽取：指标、维度、表字段、公式、口径、结论、问题、分析方法
        ↓
标准化：去重、实体对齐、别名归一、版本、权限、owner、时效
        ↓
存储：
  - 原文库
  - 结构化知识库
  - 对接已有数据目录/血缘
  - 对接已有指标口径平台
  - 全文索引
  - Milvus 向量索引
        ↓
AI 使用：
  - 知识检索
  - 指标查询
  - 表字段发现
  - SQL 生成/校验
  - 专题分析 Playbook
  - 引用证据与来源
```

## 知识来源分层

| 来源 | 典型内容 | 主要风险 | 建议处理方式 |
| --- | --- | --- | --- |
| 分析报告/周报 | 结论、图表、业务解释、异常归因 | 时效性强、结论可能过期 | 保留原文，抽取结论、指标、时间范围、业务域、引用数据。 |
| 会议纪要 | 讨论、决策、行动项、争议点 | 噪声大、未确认内容多 | 抽取决策/问题/行动项，标注 `待确认` / `已确认` / `废弃`。 |
| 指标口径文档 | 指标定义、公式、过滤条件、粒度 | 版本冲突、口径不统一 | 进入结构化指标库，设置 owner、版本、适用范围。 |
| 数据仓库元数据 | 表、字段、分区、更新频率、血缘 | 技术复杂、权限敏感 | 进入数据目录，关联 lineage、owner、质量信息。 |
| SQL/代码/ETL | 取数逻辑、计算逻辑、任务依赖 | 难以自然语言解释、版本变化 | 用 SQL parser/代码索引抽取表字段依赖和查询模板。 |
| 日常问题/踩坑 | 常见错误、修正方法、注意事项 | 来源不正式、可能只适用局部场景 | 建案例库，标注适用条件和置信度。 |
| 专题分析方法 | 固定分析思路、判断规则、多轮取数流程 | 容易混入口径细节 | 建分析 Playbook，引用指标和取数能力。 |

## 面向业务问题的知识结构

为了回答“为什么下降 / 为什么变成这样 / 怎么做得更好”，知识库不能只存文档片段。建议围绕经营分析任务组织成 6 类核心知识。

| 知识层 | 回答什么问题 | 例子 |
| --- | --- | --- |
| 指标层 | 这个指标是什么，怎么算，和哪些指标有关？ | 指标口径、公式、粒度、维度、上下游指标。 |
| 数据层 | 这个指标从哪里取，哪些表可信？ | 表、字段、SQL 模板、数据血缘、常用查询路径。 |
| 现象层 | 最近发生了什么业务变化？ | 指标下降、转化异常、流量结构变化、活动效果波动。 |
| 归因层 | 可能为什么发生？证据是什么？ | 渠道变化、供给变化、策略变更、活动节奏、外部因素。 |
| 行动层 | 接下来可以做什么？ | 优化建议、验证实验、运营动作、风险提示。 |
| 经验层 | 历史上类似情况怎么处理？ | 周报结论、复盘案例、专题分析、会议决策、踩坑。 |

这 6 层可以对应三个应用目标：

| 应用目标 | 主要依赖 |
| --- | --- |
| 解释现象 | 指标层 + 数据层 + 现象层 + 归因层 |
| 生成建议 | 归因层 + 行动层 + 经验层 |
| 支撑取数 | 指标层 + 数据层 + SQL 模板 |

## 应抽取的知识对象

建议不要只存“文档 chunk”，而是抽取成一组可治理的知识对象。

| 知识对象 | 例子 | 关键字段 |
| --- | --- | --- |
| `MetricDefinition` 指标口径 | GMV、转化率、留存率 | 指标名、别名、公式、粒度、过滤条件、口径版本、owner、状态。 |
| `DimensionDefinition` 维度定义 | 渠道、城市、用户类型 | 维度名、枚举值、来源字段、适用指标、业务解释。 |
| `DatasetAsset` 数据表资产 | dwd_xxx、ads_xxx | 表名、业务域、owner、刷新频率、分区、血缘、权限。 |
| `ColumnAsset` 字段资产 | user_id、pay_time | 字段名、类型、含义、质量、敏感级别、上下游。 |
| `SQLTemplate` 查询模板 | 某指标按天聚合 SQL | SQL、参数、适用场景、依赖表、输出字段、风险提示。 |
| `AnalysisCase` 历史分析案例 | 某次流量下滑分析 | 问题、时间范围、使用指标、结论、证据、后续验证。 |
| `AnalysisPlaybook` 专题分析流程 | 流量分析、转化分析 | 触发条件、分析步骤、取数序列、判断规则、输出结构。 |
| `DecisionRecord` 业务决策记录 | 某口径变更、某活动复盘结论 | 决策内容、背景、证据、时间、参与方、状态。 |
| `IssueAndPitfall` 问题/踩坑 | 某表延迟、某字段异常 | 问题描述、影响范围、识别方法、处理办法、有效期。 |
| `GlossaryTerm` 业务术语 | 新客、活跃、有效订单 | 定义、别名、反例、相关指标、业务域。 |
| `BusinessEvent` 业务事件 | 活动上线、策略调整、渠道变化 | 事件类型、时间、影响对象、预期影响、来源。 |
| `MetricMovement` 指标波动记录 | 某指标在某周下降 20% | 指标、时间范围、变化幅度、相关维度、初步解释。 |
| `Hypothesis` 归因假设 | 下降可能来自渠道结构变化 | 假设、证据、反证、置信度、验证方法。 |
| `ActionRecommendation` 行动建议 | 增加某渠道投放、优化某链路 | 建议、适用条件、预期影响、风险、验证指标。 |

## 标准元数据

每个知识对象和文档 chunk 都应该带元数据。否则后续无法过滤、排序、溯源和治理。

| 元数据 | 作用 |
| --- | --- |
| `source_type` | 文档、周报、会议纪要、SQL、代码、表元数据等。 |
| `source_uri` | 原文链接、代码链接、表链接、报告链接。 |
| `business_domain` | 所属业务域，如流量、交易、用户、商品、运营等。 |
| `owner` | 维护人或负责团队。 |
| `authority_level` | 权威等级：官方口径、审核报告、会议讨论、个人笔记。 |
| `status` | 草稿、待确认、已确认、已废弃、过期。 |
| `valid_from` / `valid_to` | 有效时间范围。 |
| `created_at` / `updated_at` | 创建和更新时间。 |
| `freshness` | 新鲜度，用于排序和过期提醒。 |
| `confidence` | 抽取置信度或人工确认置信度。 |
| `access_scope` | 权限范围，避免跨部门泄漏。 |
| `related_assets` | 关联指标、表、字段、报告、SQL、会议。 |
| `lineage` | 上游/下游依赖。 |

## 知识抽取流程

针对你们的场景，抽取要分两条线：

| 抽取线 | 目标 | 典型来源 |
| --- | --- | --- |
| 事实抽取 | 抽取相对稳定的事实和结构：指标、表、字段、SQL、事件。 | 指标平台、数据目录、SQL、报告事实段落。 |
| 判断抽取 | 抽取业务解释、假设、建议和经验：为什么、怎么办、是否验证。 | 周报、分析报告、会议纪要、复盘。 |

事实抽取可以更自动化；判断抽取要更谨慎，必须保留来源、时间、置信度和状态。

### 1. 采集

| 来源 | 采集方式 |
| --- | --- |
| 文档/报告 | 文档平台 API、导出、定时同步。 |
| 会议纪要 | 会议系统、IM 归档、人工上传。 |
| 数据仓库 | 元数据 API、information_schema、dbt/调度系统、BI 系统。 |
| SQL/代码 | Git、任务平台、SQL 执行日志、Notebook。 |
| 问题/踩坑 | 工单、群聊摘要、人工沉淀、Agent 反馈。 |

关键是要保存 `source_uri` 和原始快照，后续所有抽取结论都应该能回到原文。

已确认存在的系统应优先作为采集入口：

| 已有系统 | 建议定位 |
| --- | --- |
| 指标口径平台 | 指标定义、公式、口径版本的权威来源。 |
| 数据目录和血缘系统 | 表、字段、上下游、owner、刷新频率等数据资产信息的权威来源。 |
| SQL 查询体系 | 历史查询、查询模板、取数逻辑、常见参数和数据依赖的来源。 |

### 2. 解析

不同资料不要用同一种解析方法。

| 资料类型 | 解析重点 |
| --- | --- |
| PDF/Word/在线文档 | 标题层级、段落、表格、图片说明、引用。 |
| 周报/分析报告 | 结论、指标、时间范围、图表解释、异常归因。 |
| 会议纪要 | 决策、争议、行动项、负责人、状态。 |
| SQL | 依赖表、字段、过滤条件、聚合口径、参数。 |
| 代码/ETL | 输入输出表、转换逻辑、调度依赖。 |
| 表元数据 | 表/字段说明、分区、刷新频率、质量、owner。 |

Unstructured 这类工具强调先把文档拆成结构元素，再做 chunk；这比纯按字符切分更适合复杂文档。

### 3. 抽取

抽取不应该只用一种 LLM prompt。建议按知识对象拆成不同 extractor：

| Extractor | 输入 | 输出 |
| --- | --- | --- |
| 指标抽取 | 口径文档、报告、SQL | 指标名、公式、维度、过滤条件、口径版本。 |
| 数据资产抽取 | 表说明、SQL、代码 | 表、字段、业务含义、上下游。 |
| SQL 模板抽取 | 历史 SQL、查询日志 | 查询意图、参数、依赖表、输出字段。 |
| 分析结论抽取 | 周报、专题报告 | 问题、证据、结论、时间范围、适用条件。 |
| 会议决策抽取 | 会议纪要 | 决策、行动项、待确认问题、owner。 |
| 风险/踩坑抽取 | 问题记录、群聊摘要 | 错误模式、触发条件、处理办法。 |
| Playbook 抽取 | 专题分析文档、历史分析链路 | 分析步骤、取数顺序、判断规则、输出格式。 |
| 业务事件抽取 | 周报、会议纪要、报告 | 活动、策略、版本、渠道、组织动作等事件。 |
| 指标波动抽取 | 周报、报告、看板摘要 | 指标、方向、幅度、时间、维度、描述。 |
| 归因假设抽取 | 分析报告、复盘、会议讨论 | 假设、支持证据、反证、验证方式、置信度。 |
| 行动建议抽取 | 报告建议、复盘结论、会议 action | 建议、适用条件、预期收益、风险、验证指标。 |

### 3.1 周报/报告的推荐抽取模板

近期几个月的周报和报告是首批重点，建议每篇文档抽取为以下结构：

```text
ReportCard
- source_uri
- report_type: 周报 / 专题报告 / 复盘
- time_range
- business_scope
- key_metrics: 本文涉及的核心指标
- metric_movements: 指标变化记录
- business_events: 相关业务事件
- findings: 主要发现
- hypotheses: 归因假设
- evidence: 支持证据
- recommendations: 建议动作
- risks: 风险和限制
- open_questions: 未解决问题
```

其中最关键的是把“事实”和“判断”分开：

| 类型 | 例子 | 存储要求 |
| --- | --- | --- |
| 事实 | 指标 A 本周下降 15%；渠道 B 流量减少。 | 可结构化入库，关联指标和时间。 |
| 判断 | 下降可能是渠道 B 供给减少导致。 | 标为 hypothesis，必须保留证据和置信度。 |
| 建议 | 建议增加某渠道投放或优化某链路。 | 标为 recommendation，保留适用条件和验证指标。 |

### 3.2 会议纪要的推荐抽取模板

会议纪要从现在开始积累，建议不要原封不动全部当知识使用，而是抽取：

```text
MeetingCard
- source_uri
- meeting_time
- participants / teams
- discussed_topics
- decisions
- action_items
- business_events
- hypotheses
- open_questions
- status: 待确认 / 已确认 / 已废弃
```

会议纪要默认权威等级应低于正式报告和指标平台，除非某条决策被确认。

### 4. 标准化与实体对齐

抽取出来的知识必须归一，否则会越来越乱。

| 问题 | 处理方式 |
| --- | --- |
| 同一个指标多个名字 | 建 alias 表，映射到 canonical metric id。 |
| 同一个表被多处描述 | 合并到同一个 dataset asset，保留来源证据。 |
| 新旧口径冲突 | 按版本、有效期、authority_level 判断，不直接覆盖。 |
| 会议讨论和正式口径冲突 | 默认正式口径优先，会议内容标成待确认或历史讨论。 |
| SQL 和文档口径不一致 | 标记冲突，进入人工 review 队列。 |

### 5. 校验

经营分析知识库必须有校验环节。

| 校验对象 | 校验方式 |
| --- | --- |
| 指标公式 | 与指标平台/语义层/官方口径对齐。 |
| 表字段 | 与数仓元数据和实际 schema 对齐。 |
| SQL 依赖 | SQL parser、执行计划、血缘系统校验。 |
| 报告结论 | 保留来源和时间范围，不直接升级为永恒事实。 |
| 会议决策 | 需要 owner 或业务负责人确认。 |

考虑到知识确认机制还不明确，可以先分级处理：

| 知识类型 | 第一阶段处理 |
| --- | --- |
| 指标口径、表字段、血缘 | 直接以已有平台/API 为准。 |
| 报告中的指标变化事实 | 可自动入库，但必须带来源和时间范围。 |
| 报告中的归因和建议 | 可入库为 hypothesis / recommendation，不作为确定事实。 |
| 会议纪要中的讨论 | 默认待确认，仅作为参考证据。 |
| 会议纪要中的明确决策 | 需要后续设计确认机制后再提升权威等级。 |

## 存储设计

建议采用“多存储、多索引”的架构。

| 层 | 存什么 | 适合技术 |
| --- | --- | --- |
| 原文层 | 原始文档、报告、会议纪要、SQL、代码快照 | 对象存储、文档库、Git、文档平台。 |
| 文档结构层 | document、section、element、chunk | PostgreSQL / 文档数据库。 |
| 结构化知识层 | 指标、表、字段、SQL 模板、分析案例、Playbook | PostgreSQL / MySQL。 |
| 元数据/血缘层 | 数据资产、owner、tag、glossary、lineage | 优先对接已有数据目录/血缘系统。 |
| 指标/语义层 | metric、dimension、entity、semantic model | 优先对接已有指标口径平台。 |
| 向量索引层 | chunk embedding、对象 embedding | Milvus。 |
| 全文索引层 | 文档、SQL、字段名、指标名关键词检索 | Elasticsearch / OpenSearch / SQLite FTS / 其他搜索引擎。 |
| 图关系层 | 指标-表-字段-报告-会议-案例之间的关系 | 图数据库或关系库中的边表。 |

Milvus 的位置：

```text
Milvus = 向量召回层
不是 = 原文库 / 指标口径库 / 权限系统 / 血缘系统 / 唯一事实源
```

### 推荐的逻辑数据模型

如果不先绑定具体数据库，可以先按逻辑表/对象设计：

| 对象 | 作用 |
| --- | --- |
| `documents` | 原始文档/报告/会议纪要元信息。 |
| `chunks` | 文档切片，进入向量知识库和全文检索。 |
| `metrics` | 从指标平台同步或映射来的指标。 |
| `datasets` / `columns` | 从数据目录同步来的表和字段。 |
| `sql_templates` | 历史可复用 SQL 或查询模板。 |
| `report_cards` | 每篇周报/报告的结构化卡片。 |
| `meeting_cards` | 每次会议的结构化卡片。 |
| `metric_movements` | 指标波动事实。 |
| `business_events` | 业务事件。 |
| `hypotheses` | 归因假设。 |
| `recommendations` | 行动建议。 |
| `analysis_cases` | 历史分析案例。 |
| `playbooks` | 专题分析流程。 |
| `relations` | 指标、表、文档、事件、假设、建议之间的关系。 |

向量知识库里不一定只放 raw chunk，也可以放部分结构化对象的文本表示：

| 入向量库的对象 | 用途 |
| --- | --- |
| 文档 chunk | 报告问答、查证据。 |
| ReportCard 摘要 | 快速找到相关报告。 |
| AnalysisCase 摘要 | 找历史类似案例。 |
| Hypothesis 摘要 | 找类似归因。 |
| Recommendation 摘要 | 找历史建议和动作。 |
| Playbook 摘要 | 找适用分析流程。 |

但指标口径、表字段、血缘这类高精度信息，应优先走 API/结构化查询，而不是只靠向量相似度。

## 给 AI 的交付方式

虽然你说这块不急，但存储设计必须为 AI 使用方式服务。

Agent 不应该只有一个“知识库搜索工具”，而应该有多个专用工具：

| AI 工具 | 作用 |
| --- | --- |
| `metric_lookup` | 查指标口径、公式、版本、owner、适用范围。 |
| `dataset_discovery` | 找表、字段、业务含义、刷新频率、权限。 |
| `lineage_lookup` | 查指标/表/字段上下游依赖。 |
| `sql_template_search` | 找历史 SQL 和查询模板。 |
| `analysis_case_search` | 找历史类似分析案例。 |
| `playbook_get` | 获取专题分析流程，如流量分析、转化分析。 |
| `document_search` | 查原始报告、周报、会议纪要。 |
| `knowledge_conflict_check` | 检查口径冲突、过期、未确认状态。 |
| `metric_movement_search` | 查某指标近期/历史波动记录。 |
| `business_event_search` | 查相关业务事件、活动、策略变更。 |
| `hypothesis_search` | 查历史归因假设和证据。 |
| `recommendation_search` | 查历史建议和行动效果。 |

给模型的上下文应该是：

```text
用户问题
+ 结构化事实：指标/表/字段/口径
+ 检索证据：报告/文档/会议片段
+ 数据约束：权限/时间范围/可用表
+ 分析方法：Playbook
+ 风险提示：过期、冲突、置信度
+ 来源引用：source_uri / line range / SQL link
```

## 针对经营分析的典型任务流

### 任务 0：老板追问“为什么下降”

```text
用户问：这个指标为什么大幅下降？
-> metric_lookup 查指标定义和相关维度
-> metric_movement_search 找近期/历史波动
-> dataset_discovery / SQL 查询拿实际数据
-> business_event_search 找同期事件
-> analysis_case_search 找历史类似案例
-> hypothesis_search 形成候选归因
-> 验证/反证主要假设
-> 输出：现象、可能原因、证据、置信度、下一步验证和建议
```

### 任务 0.1：老板追问“怎么做得更好”

```text
用户问：能不能给点建议？
-> playbook_get 找对应分析框架
-> recommendation_search 找历史动作
-> analysis_case_search 找类似成功/失败案例
-> 结合当前数据和约束
-> 输出：建议列表、优先级、预期影响、风险、验证指标
```

### 任务 1：只查指标口径

```text
用户问指标
-> metric_lookup
-> alias 归一
-> 返回官方口径、公式、粒度、过滤条件、版本、owner、来源
```

### 任务 2：取数

```text
用户描述需求
-> 澄清指标/维度/时间/口径
-> metric_lookup 找口径
-> dataset_discovery 找表字段
-> sql_template_search 找历史写法
-> 生成 SQL / 调用取数工具
-> 校验结果和口径
```

### 任务 3：专题分析

```text
用户提出分析主题
-> playbook_get 获取分析流程
-> 分解多个取数步骤
-> 每一步查口径/表/SQL
-> 结合历史 case 和报告
-> 生成分析判断
-> 附来源、口径、数据范围、风险提示
```

### 任务 4：复用历史经验

```text
用户问类似问题
-> analysis_case_search
-> 找相似历史案例
-> 对齐业务域、时间范围、指标
-> 判断是否可复用
-> 返回案例结论和适用条件
```

## 质量评估

知识库效果不能只看“搜出来像不像”，要建立业务评估集。

| 评估项 | 示例指标 |
| --- | --- |
| 召回准确性 | 正确知识是否出现在 top-k。 |
| 来源正确性 | 回答引用的来源是否权威、是否最新。 |
| 指标口径正确性 | 是否使用正确公式、粒度、过滤条件。 |
| SQL 正确性 | SQL 是否能跑、是否查对表、是否符合口径。 |
| 权限正确性 | 是否避免越权引用。 |
| 时效正确性 | 是否避免使用过期报告或旧口径。 |
| 分析质量 | 结论是否由数据支持，是否区分事实和判断。 |
| 冲突识别 | 遇到多口径、多版本时是否提示冲突。 |

建议沉淀一批真实问题作为 golden set：

```text
指标口径问题 50 个
取数问题 50 个
表字段发现问题 50 个
专题分析问题 20 个
历史案例复用问题 20 个
冲突/过期/权限问题 20 个
```

## 落地路线

### Phase 0：范围确认

目标：明确已有系统边界和第一批任务的最小闭环。由于专题分析、报告问答、查口径、取数都要覆盖，Phase 0 的重点不是缩成单一场景，而是定义每类任务的最小可用能力。

需要确认：

1. 第一批业务域是什么。
2. 指标口径平台、数据目录/血缘系统、SQL 查询体系分别提供哪些 API 或导出方式。
3. Milvus 是否确定使用，以及当前是否已有 collection / schema / embedding 模型。
4. 报告问答和专题分析的首批文档范围。
5. 知识确认机制暂不明确，需要先定义哪些内容可以自动入库，哪些必须人工 review。

### Phase 1：最小可用知识库

优先做：

| 模块 | 目标 |
| --- | --- |
| 指标口径库 | 能查 50-100 个核心指标。 |
| 表字段目录 | 覆盖核心业务域的关键表。 |
| 文档检索 | 接入核心报告/周报。 |
| Milvus 向量检索 | 支持语义召回文档 chunk 和案例。 |
| 全文检索 | 支持表名、字段名、指标名、SQL 关键词。 |
| 来源引用 | 每个回答能回到原文。 |
| 基础任务工具 | 覆盖 `metric_lookup`、`document_search`、`dataset_discovery`、`sql_template_search`。 |
| 报告结构化抽取 | 把近期几个月周报/报告抽成 `ReportCard`、`MetricMovement`、`Hypothesis`、`Recommendation`。 |
| 经营问题问答闭环 | 覆盖“为什么下降”“为什么变成这样”“怎么做得更好”三类问题。 |

### Phase 2：经营分析能力增强

增加：

| 模块 | 目标 |
| --- | --- |
| SQL 模板库 | 复用历史查询和计算逻辑。 |
| 分析案例库 | 复用历史专题分析。 |
| Playbook 库 | 支持流量分析、转化分析等固定流程。 |
| 数据血缘 | 指标、表、字段、报表之间能追溯。 |
| 冲突检测 | 标出口径冲突和过期知识。 |

### Phase 3：治理与自动化

增加：

| 模块 | 目标 |
| --- | --- |
| 自动增量同步 | 文档、SQL、表元数据变更后自动更新。 |
| 人工 review 工作台 | 处理低置信度抽取、冲突、过期知识。 |
| 质量评估集 | 定期评估 Agent 检索和回答质量。 |
| 反馈闭环 | 用户纠错后进入知识修正流程。 |
| 权限审计 | 记录 AI 使用了哪些知识来源。 |

## 待确认问题

为了后续细化，需要确认这些点：

| 问题 | 为什么重要 |
| --- | --- |
| 这个小业务方向的核心指标列表是什么？ | 决定首批 `MetricMovement` 和归因链路。 |
| 老板最常问的 20-50 个问题是什么？ | 决定知识抽取和评估集的优先级。 |
| 指标口径平台能否提供稳定 API？ | 决定是实时查询平台，还是定时同步成 AI 知识对象。 |
| Milvus 目前存的是什么粒度？文档 chunk、指标对象，还是表字段对象？ | 决定是否需要重建索引。 |
| 是否已有 Elasticsearch/OpenSearch/其他全文检索？ | 决定关键词检索如何实现。 |
| 数据目录/血缘系统能否被 Agent 调用？ | 决定是作为在线工具调用，还是离线同步一份。 |
| 文档来源在哪些系统？ | 决定采集器和权限同步方式。 |
| 哪些知识必须人工确认后才能给 AI 使用？ | 决定 review workflow；当前尚不明确。 |
| 输出是给运营同学看，还是给 Agent 自动执行取数？ | 决定准确性和校验强度。 |
| 周报/报告里的建议是否有后续效果记录？ | 决定能否做“建议有效性”学习，而不只是复述历史建议。 |

## 参考资料要点

| 资料 | 对本方案的启发 |
| --- | --- |
| LlamaIndex ingestion pipeline | 文档进入 RAG 前需要 loaders、transformations、docstore、去重和增量管理。 |
| LangChain document loaders / splitters | 多来源文档需要标准 loader；chunk 策略要按内容类型调整。 |
| Unstructured chunking | 复杂文档应先解析为结构元素，再 chunk，而不是纯字符切。 |
| Haystack Document Store / Retriever | 存储和检索器是分离概念，不同存储适合不同 retriever。 |
| DataHub lineage | 数据血缘用于理解数据从哪里来、如何流动、影响哪些下游。 |
| OpenMetadata glossary | 业务术语和 glossary 可以作为数据资产标签，帮助发现、检索和治理。 |
| dbt Semantic Layer / MetricFlow | 指标定义应该集中管理，保证下游工具和应用使用一致口径。 |
